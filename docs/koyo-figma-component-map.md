## Summary

The Figma file is a shadcn-style design system with top-level pages for primitives such as Accordion, Alert, Avatar, Badge, Button, Card, and Input & Input Group. Each inspected component page contains a first component area that appears to be the base/store reference and a second same-named component area that should be treated as the Koyo-owned baseline. The base/store component is useful for anatomy, naming, variants, and behavior reference only; docs and implementation planning should prefer the second component area. Migration means visual/token/component parity with the Koyo component layer; renaming shadcn primitives or only changing copy does not count.

The linked node is the first Badge frame on the Badge page (`273:30617`). It contains `.Component Page Header`, `.ProWarning`, `Component`, and `Examples` sections. The upstream/base-store-ish Obra/community warning and docs link in that page are reference/legacy context only, not Koyo output. The Koyo Badge baseline is the next component area on that page: the first `Brand Theme` frame (`3455:13168`), which contains the second `Badge` component set (`3455:13174`).

Figma inspection used available connector metadata, screenshots for the linked Badge node, and read-only plugin API scripts. Main-thread `get_design_context` for Badge (`273:30617`) also confirmed Badge variants and exact token values. The dedicated `get_design_context` tool was not exposed in this sub-session, so non-Badge token and variant details below come from component metadata and bound-variable inspection rather than a full design-context payload.

## Structure Rule

- Base/store reference: first same-named component frame on a component page, usually positioned at `x=0`.
- Koyo-owned baseline: second same-named component frame or first subsequent themed frame on the same page.
- Extra copies: later `Brand Theme` or same-named frames are likely theme experiments or duplicates. Treat them as secondary references unless a designer explicitly promotes one.
- Docs baseline: document the Koyo-owned baseline first; mention base/store only where it explains anatomy or expected behavior.

## Confidence Notes

- High confidence: page names, component set names, component property definitions, variant options, and token names listed above came directly from Figma connector inspection.
- High confidence: Badge variants, booleans, and exact token values were additionally confirmed by main-thread `get_design_context` for node `273:30617`.
- Medium confidence: base/store versus Koyo-owned classification is inferred from the user's rule plus repeated page structure: first component area as reference, second component area as Koyo baseline.
- Medium confidence: later `Brand Theme` frames on Badge and later duplicate Button/Accordion frames appear to be theme copies or experiments. They should not override the second component area without designer confirmation.
- Partial data: only the linked Badge node had a screenshot captured during this pass. Other primitives were inspected structurally through metadata and plugin API scripts.
- Partial data: `get_design_context` was not available in the exposed Figma tools for this sub-session; exact rendered spacing/color values for non-Badge primitives should be revalidated when implementing.
- Partial data: Select, Checkbox, Switch, and Radio Group docs can state the implemented Koyo selected/focus/error token layer, but should not imply exact Figma node parity until the Koyo-owned Figma components are inspected.

## Next-Step Checklist For `docs/catalog`

- Create a `docs/catalog` entry per primitive using the Koyo component IDs above as the baseline source.
- For each entry, document Figma variants as public API props before writing examples.
- Map Figma `Roundness` to a code prop only where the implementation supports both default and pill/round styles.
- Map Figma interactive `State` values to CSS selectors or documented behavior, not user-facing props, unless the docs need forced-state examples.
- Keep icon booleans and instance swaps as implementation anatomy: `leftIcon`, `rightIcon`, and slot examples for Button, Badge, Alert, and Input.
- Reconcile Figma tokens with local CSS variables before documenting hard values; prefer `--primary`, `--secondary`, `--destructive`, `--border`, `--input`, `--card`, and typography/radius utilities where they already exist.
- For Badge, start from the confirmed exact values above, but still express docs in Koyo tokens instead of copying Obra/community page language.
- Add examples that demonstrate the Koyo baseline first, then include base/store-inspired edge cases only if they are supported in Koyo UI.
- Re-open Figma for screenshots or full design context before final visual QA, especially for Button, Input, and Card where theme duplicates exist.
